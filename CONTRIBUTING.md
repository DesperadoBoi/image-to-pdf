# CONTRIBUTING

## Базовые правила

- ветка `main` должна оставаться рабочей;
- напрямую в `main` не коммитить;
- каждая задача выполняется в отдельной ветке;
- один разработчик или один AI-агент отвечает за одну ветку;
- два агента не должны одновременно изменять одну ветку;
- одна ветка должна решать одну логическую задачу;
- изменения попадают в `main` через Pull Request;
- перед merge второй участник просматривает diff;
- конфликтующие архитектурные решения сначала обсуждаются и фиксируются в документации.

## Именование веток

- `feature/<short-name>`;
- `fix/<short-name>`;
- `chore/<short-name>`;
- `docs/<short-name>`;
- `test/<short-name>`.

Примеры:

- `feature/image-picker`;
- `feature/pdf-generator`;
- `feature/page-editor`;
- `fix/exif-orientation`;
- `chore/project-foundation`.

## Обычный старт задачи

```powershell
git switch main
git pull --ff-only
git switch -c feature/example
```

## Перед Pull Request

- проверить `git diff`;
- убедиться, что нет секретов и локальных файлов;
- выполнить сборку;
- выполнить тесты;
- выполнить `lint`;
- обновить документацию, если изменились продуктовые или архитектурные решения;
- сделать тематический commit;
- отправить ветку;
- создать Pull Request.

## Commit messages

- писать кратко на русском языке;
- один commit должен иметь одну понятную цель;
- не использовать сообщения `update`, `changes`, `fix`, `work`.

Хорошие примеры:

- `Добавлен выбор изображений`
- `Реализована генерация PDF`
- `Исправлена обработка EXIF`
- `Добавлены правила разработки`

## Конфликты

- не использовать `git reset --hard`;
- не использовать `force-push` без согласования;
- перед разрешением конфликта обновить локальную `main`;
- внимательно проверять обе стороны конфликта;
- после разрешения повторно запускать проверки.
